Binary Trees — The Foundation of Heaps

A heap is built on a special type of binary tree. Before understanding heaps, we need to nail down a few key definitions from the OSU slides that are commonly tested.

Key Definitions

Binary Tree

A rooted tree where each node has at most 2 children (left and right). Nodes with no children are leaf nodes.

Height
h = log n

Number of edges on the longest path from root to a leaf.

Complete Binary Tree

All internal nodes have exactly 2 children AND all leaves are at the same depth.

Nearly Complete

All levels completely filled except the last, which fills left-to-right. This is what heaps use!

Theorem: A complete binary tree of height h has 2^(h+1) − 1 nodes. Corollary: height ≈ log n. This is why heap operations are O(log n)!

Array Index Formulas (Memorize These!)

Heaps store the tree in an array, 1-indexed. These three formulas navigate the tree:

Parent of node i
⌊i/2⌋

Node 5's parent is at index 2

Left child of i
2i

Node 3's left child is at index 6

Right child of i
2i+1

Node 3's right child is at index 7

What is a Heap?

A heap is a nearly complete binary tree stored in an array that satisfies the heap property. There are two flavors: max-heap and min-heap.

Max-Heap Property

Every parent ≥ its children.
Root = maximum element in the entire heap.

Min-Heap Property

Every parent ≤ its children.
Root = minimum element in the entire heap.

Common Misconception: A heap is NOT a sorted array. Nodes on the same level have no guaranteed ordering relative to each other — only parent/child relationships matter.

Heap in Array Form

The beauty of a heap: no pointers needed! The tree structure is entirely encoded by index arithmetic.

Index 0 is unused (1-indexed). Parent(i) = ⌊i/2⌋, Left(i) = 2i, Right(i) = 2i+1

Inserting into a Max-Heap (Heapify-Up)

Insertion adds the new key at the end of the array, then bubbles it up by swapping with its parent until the heap property is restored.

The Algorithm

MaxHeapInsert(A[], size, K)
  1: size ← size + 1
  2: i ← size
  3: A[i] ← K   // Place at end
  4: while (i > 1) and (A[Parent(i)] < A[i]) do
  5:     Swap(A[i], A[Parent(i)])   // Bubble up
  6:     i ← Parent(i)
  7: end while
Best Case
O(1)

New key is ≤ its parent — no swaps needed

Worst Case
O(log n)

New key bubbles all the way to root (height of tree)

Extracting the Maximum (Heapify-Down)

Extraction removes the root (maximum), replaces it with the last element, shrinks the heap, then sifts down to restore the heap property.

Extract-Max Algorithm

HeapExtractMax(A[], size)
  1: if size < 1 then error "underflow"
  2: maxKey ← A[1]   // Save the max
  3: A[1] ← A[size]   // Move last to root
  4: size ← size − 1
  5: MaxHeapify(A, size)   // Sift down
  6: return maxKey

MaxHeapify (Sift Down)

MaxHeapify(A[], size)
  j ← 1; flagDone ← false
  repeat
    L ← Left(j); R ← Right(j)
    largest ← j
    if L ≤ size and A[L] > A[largest] then largest ← L
    if R ≤ size and A[R] > A[largest] then largest ← R
    if j ≠ largest then Swap(A[j], A[largest]); j ← largest
    else flagDone ← true
  until flagDone = true
Worst Case (both)
O(log n)

Sift down travels entire height of tree

Key Insight

Largest among {parent, left, right} wins. Always swap downward to maintain max-heap.

Heap Sort & Priority Queues

Two major applications of heaps: sorting in O(n log n) and implementing a priority queue efficiently.

HeapSort

HeapSort(B[], n)
  // Phase 1: Build heap by inserting all elements
  size ← 0
  for i ← 1 to n do
    MaxHeapInsert(A, size, B[i])
  end for
  // Phase 2: Extract max n times → sorted order
  for i ← n downto 1 do
    B[i] ← HeapExtractMax(A, size)
  end for
Complexity: Phase 1 is O(n log n), Phase 2 is O(n log n). Total: O(n log n) worst case.

Priority Queue Comparison (Exam Favorite!)

The OSU slides give this table explicitly — know it cold:

Operation Heap (Binary) Unsorted Array Sorted Array
Insert O(log n) O(1) O(n)
Extract-Max O(log n) O(n) O(1)
Sort (via PQ) O(n log n) O(n²) O(n²)
Key Insight: The heap wins because BOTH insert and extract are O(log n). Arrays give O(1) for one but O(n) for the other — making sorting O(n²). The heap's balance is what makes it powerful.

Heap Increase-Key

MaxHeapIncreaseKey(A[], size, i, K)
  if K < A[i] then error "new key < current"
  A[i] ← K
  while (i > 1) and (A[Parent(i)] < A[i]) do
    Swap(A[i], A[Parent(i)]); i ← Parent(i)
  end while

Direct-Address Tables & Hash Tables

Hash tables solve the problem of storing data when we can't afford an array the size of the entire key universe.

Direct Address

Element with key k → slot k.
Fast, but needs array of size |U| (universe). Wasteful if few keys used.

Hash Table

Element with key k → slot h(k).
Table size m ≪ |U|. Space-efficient!

The Hash Function

A hash function maps the universe U of keys into slots {0, 1, ..., m−1}:

h : U → {0, 1, ..., m − 1}
Key k hashes to slot h(k). The value h(k) is the hash value of k.

Collision Resolution: Chaining

A collision occurs when two keys hash to the same slot. Chaining resolves this by storing all colliding elements in a linked list at each slot.

How Chaining Works

Each slot j holds a pointer to a linked list of all keys that hash to j. If no keys hash to j, slot j = NIL.

INSERT(T, x): insert x at head of T[h(x.key)]  // O(1)
SEARCH(T, k): search list T[h(k)] for key k  // O(list length)
DELETE(T, x): delete x from list T[h(x.key)]  // O(1) with doubly linked

Performance Analysis

Let α = n/m be the load factor (elements per slot on average).

CaseSearch TimeWhen?
Worst Case O(n) All n keys hash to same slot
Expected (avg) Θ(α + 1) Uniform hash assumption
Practical (α = O(1)) O(1) m proportional to n
Load Factor α = n/m: If m grows with n (keep α ≤ 1), all operations average O(1). This is the magic of hash tables!

Hash Functions — Three Types

The OSU slides cover three specific hash functions. Know each formula and its properties.

Properties of a GOOD Hash Function

1. Uniformity — each key has equal probability of hashing to any slot
2. Determinism — same key always → same hash value
3. Minimal collisions — minimizes keys mapping to same slot

1. Division Hashing

h(key) = key mod m   // or mod p (prime)

Simple and fast. Works well if m is prime and not close to a power of 2. Poor if many keys share the same remainder mod m.

2. Multiplicative Hashing

h(k) = ⌊m × frac(kσ)⌋   // σ ∈ [0.5, 1]

frac(x) is the fractional part of x. Multiply key by constant σ, extract the fraction, scale by m. More robust than division hashing.

3. Universal Hashing

h(k) = ((ak + b) mod p) mod m
// a,b random in {1,...,p-1}, p prime, p ≫ m
Why Universal? Randomly choosing a, b makes it hard for an adversary to pick keys that all collide. Each key has equal probability 1/m of hashing to any slot, regardless of key distribution.

Open Addressing — No Linked Lists!

In open addressing, all elements live inside the table itself. When a collision occurs, we probe for the next empty slot.

Chaining

Elements stored in linked lists outside the table. Table can store > m elements (α > 1 OK).

Open Addressing

All elements stored inside the table. Table fills up — requires α < 1 always.

The Probe Sequence

We extend h to include a probe number: h : U × {0,...,m−1} → {0,...,m−1}

For each key, probe sequence (h(k,0), h(k,1), ..., h(k,m−1)) must cover all slots.

Three Probing Strategies

Linear probing:    h(k,j) = (h(k) + j) mod m
Quadratic probing: h(k,j) = (h(k) + c₁j + c₂j²) mod m
Double hashing:   h(k,j) = (h₁(k) + j·h₂(k)) mod m

Insert Algorithm

HashTable.Insert(K)
  i ← 0
  repeat
    j ← h(k, i)
    if T[j] = Null then
      T[j] ← k; return j
    else i ← i + 1
  until i = m  // ERROR: overflow

Performance

Expected runtime: O(1 / (1 − α)) where α = n/m
If n ≤ m/2 (α ≤ 0.5), then expected time → O(1). Keep your table at most half-full!
